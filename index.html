Version Control

Vesrion Control (aka Source Control, Revision Control) let's you track your files over time
Version contol is like having a giant Undo button for your project
It makes it easy to save versions of your files at different points in time
you can also restore previous versions and compare different versions
it also makes it a lot easier to collaborate with other people (No more .zip files)
Version Control examples: CVS (Concurrent Version Systems) , SVN (Subversion), Mercurial (hg) and Git
Handling files without Version Control?

A Good VCS does the following

  • Backup and Restore.
  • Synchronization.
  • Short-term undo.
  • Long-term undo.
  • Track Changes.
  • Track Ownership.
  • Sandboxing
  • Branching and merging.

Git: Installing

Git Repository

When you work on a project, you'll often have multiple files that you want to track together.
Git calls such a collection of files a repository.

Git clone

Git has a command allowing you to copy an entire repository from one computer to another
The command is called git clone because it creates a clone of the repository.
Git can work offline, so after cloning a repository, you don't actually need an internet connection

Git Commits

Git allows user to create checkpoints of the code/files manually
Git's system makes it much more likely that each individual version represent a logical change.
In Git terminology, these user-created checkpoints are called commits.
Commits are the basic building blocks of Git, each one representing a version of the content at one point in time.
A commit is a snapshot of every file in your repository at the time you choose to make the commit.
So for example, this is the fourth commit in a repository in which you have files a, b, and c If you've only changed file a in this fourth commit, then files b and c will be identical between commits 4 and 3
Git requires the user to supply commit message each time a commit is created.
This makes looking through old versions much more useful, assuming that the user supplied a useful commit message.

Git Checkout

we can also temporarily change our files back to how they were at the time of any commit.
This is called a git checkout
In git, checking out a commit means resetting all of your files to how they were at the time that commit was made.
One reason to do a checkout might be, if a bug was introduced, but you're not sure which commit introduced it.

A little of details

Git has three Stages: working directory, staging area, and the git directory
The Git directory (aka repositroy) is where Git stores the metadata and object database for your project.
This is the most important part of Git, and it is what is copied when you clone a repository from another computer.
The working tree is a single checkout of one version of the project. These files are pulled out of the compressed database in the Git directory and placed on disk for you to use or modify.
The staging area is a file, generally contained in your Git directory, that stores information about what will go into your next commit. It’s sometimes referred to as the “index”, but it’s also common to refer to it as the staging area.

Making a git repository

Repositories are just normal directories on your machine
The only real difference is that git repositories store a bunch of metadata about the history of the repository since it was created.
You can either run this in an empty directory and create files later or create the repository in a directory that already contains files.
Git by default does not add any files in the tracked folder
This is because you might not want to commit everything in the directory, and
Another reason may be that you wouldn't be able to write your own commit message if Git created the commit for you.
